# E2E 测试指南 - 多角色认证与授权系统

**RFC**: 0010  
**创建日期**: 2025-12-06  
**最后更新**: 2025-12-06

---

## 📋 测试概览

本指南提供手动端到端（E2E）测试步骤，验证多角色认证与授权系统的所有功能。

### 测试环境要求

- ✅ 本地开发环境运行（`npm run dev`）
- ✅ 或生产环境（Vercel 部署）
- ✅ Supabase 数据库已部署迁移
- ✅ 至少 3 个测试用户账户（student, admin, owner）

---

## 🧪 测试场景 1: Student 角色测试

### 准备

1. 创建一个 student 用户（或使用现有用户）
2. 确保用户只有 `student` 角色

### 测试步骤

#### 1.1 登录和基本访问

1. **登录**
   - 访问 `/sign-in`
   - 使用 student 账户登录
   - ✅ 应该成功登录并重定向到 `/dashboard`

2. **访问学生页面**
   - 访问 `/dashboard` → ✅ 应该可以访问
   - 访问 `/learning` → ✅ 应该可以访问
   - 访问 `/practice` → ✅ 应该可以访问
   - 访问 `/lessons` → ✅ 应该可以访问
   - 访问 `/profile` → ✅ 应该可以访问
   - 访问 `/settings` → ✅ 应该可以访问

3. **访问受限页面（应该被拒绝）**
   - 直接访问 `/admin` → ❌ 应该被重定向到 `/dashboard`
   - 直接访问 `/owner` → ❌ 应该被重定向到 `/dashboard`
   - 直接访问 `/tutor` → ❌ 应该被重定向到 `/dashboard`

#### 1.2 角色检查

1. **检查用户角色**
   - 在浏览器控制台执行：
     ```javascript
     // 检查 store 中的角色
     // 应该返回 ['student']
     ```
   - ✅ 应该只显示 `student` 角色

2. **检查角色方法**
   - `hasRole('student')` → ✅ 应该返回 `true`
   - `hasRole('admin')` → ❌ 应该返回 `false`
   - `hasAnyRole(['admin', 'owner'])` → ❌ 应该返回 `false`

## 🧪 测试场景 2: Admin 角色测试

### 准备

1. 使用 owner 账户将测试用户提升为 `admin` 角色
2. 或直接在数据库中执行：
   ```sql
   SELECT update_user_role(
     '<user-id>'::UUID,
     'admin',
     'E2E 测试：提升为管理员'
   );
   ```

### 测试步骤

#### 2.1 登录和基本访问

1. **登录**
   - 使用 admin 账户登录
   - ✅ 应该成功登录

2. **访问管理员页面**
   - 访问 `/admin` → ✅ 应该可以访问
   - 访问 `/admin/users` → ✅ 应该可以访问
   - 访问 `/dashboard` → ✅ 应该可以访问

3. **访问受限页面**
   - 直接访问 `/owner` → ❌ 应该被重定向到 `/admin`
   - 直接访问 `/tutor` → ✅ 如果 admin 也有 tutor 角色，可以访问；否则被重定向

#### 2.2 用户管理功能（只读）

1. **查看所有用户**
   - 访问 `/admin/users`
   - ✅ 应该显示所有用户列表
   - ✅ 应该显示每个用户的角色

2. **尝试修改用户角色（应该失败）**
   - 在 UI 中尝试修改用户角色
   - ❌ 应该显示错误："只有超级用户可以添加 admin 或 owner 角色"
   - 或 UI 中不显示修改按钮（只读模式）

#### 2.3 角色检查

1. **检查用户角色**
   - ✅ 应该显示 `admin` 角色（可能还有 `student` 角色）

2. **检查角色方法**
   - `hasRole('admin')` → ✅ 应该返回 `true`
   - `hasRole('owner')` → ❌ 应该返回 `false`
   - `hasAnyRole(['admin', 'owner'])` → ✅ 应该返回 `true`

## 🧪 测试场景 5: 审计日志测试

### 测试步骤

#### 5.1 角色变更记录

1. **执行角色变更**
   - 使用 owner 账户修改用户角色
   - ✅ 应该成功

2. **检查审计日志**
   - 在数据库中查询：
     ```sql
     SELECT * FROM public.role_changes
     ORDER BY created_at DESC
     LIMIT 10;
     ```
   - ✅ 应该看到最新的角色变更记录
   - ✅ 应该包含：user_id, old_role, new_role, changed_by, reason, created_at

#### 5.2 审计日志完整性

1. **尝试直接插入审计日志（应该失败）**
   - 在 Supabase Dashboard → SQL Editor 中执行：
     ```sql
     INSERT INTO public.role_changes (user_id, new_role, changed_by)
     VALUES ('<some-user-id>'::UUID, 'admin', '<some-user-id>'::UUID);
     ```
   - ❌ 应该失败（RLS 策略禁止直接插入）

## 🧪 测试场景 7: 错误处理测试

### 测试步骤

#### 7.1 错误代码系统

1. **测试无效角色**
   - 尝试添加无效角色（例如：'invalid_role'）
   - ❌ 应该返回错误代码 `P1202` (VALIDATION_INVALID_ROLE)
   - ✅ 错误消息应该由 UI 显示（不是数据库返回的原始消息）

2. **测试权限不足**
   - 使用 admin 账户尝试添加 `admin` 角色
   - ❌ 应该返回错误代码 `P1104` (PERMISSION_REQUIRES_OWNER_FOR_ROLE)
   - ✅ 错误消息应该由 UI 显示

3. **测试用户不存在**
   - 尝试修改不存在的用户角色
   - ❌ 应该返回错误代码 `P1201` (VALIDATION_TARGET_USER_NOT_FOUND)

## 🔧 故障排除

### 问题：无法访问页面

1. 检查用户是否已登录
2. 检查用户角色是否正确
3. 检查路由权限配置（`route_role_permissions` 表）
4. 检查浏览器控制台错误

### 问题：角色修改失败

1. 检查当前用户是否有权限（admin 或 owner）
2. 检查目标用户是否存在
3. 检查错误代码和消息
4. 检查数据库函数日志

### 问题：审计日志未记录

1. 检查 `role_changes` 表是否存在
2. 检查 RLS 策略是否正确
3. 检查数据库函数是否正常执行
